# Архитектура Otragenie Camp

## Обзор

Проект состоит из трёх основных частей:

1. фронтенд-лендинг на React/Vite;
2. backend на Express;
3. Telegram-бот на Telegraf с SQLite-хранилищем.

Все части запускаются из одного процесса через `server.ts`.

## Контур приложения

```text
Пользователь сайта
  -> React frontend
    -> AI чат Gemini
    -> формы заявок
    -> запросы к Express API

Express API
  -> Telegram sendMessage
  -> Prodamus payment URL
  -> Prodamus webhook

Telegram Bot
  -> личный чат пользователя
  -> forum topic в группе
  -> SQLite bot.db
```

## Фронтенд

### Основные файлы
- `src/App.tsx` - основная страница, модальные окна, CTA, тарифы, секции контента.
- `src/data.ts` - централизованный контент лендинга.
- `src/components/ChatAssistant.tsx` - встроенный AI-чат.
- `src/index.css` - тема, типографика, цвета и utility-слои.

### Роль фронтенда

Фронтенд выполняет две задачи:
- продаёт участие в кэмпе через лендинг;
- даёт посетителю быстрый канал общения через AI-чат и формы.

### Особенности

- `App.tsx` очень крупный и совмещает UI, контент и часть поведения модалок.
- Значительная часть контента хранится локально в `src/data.ts`, без CMS.
- Для анимаций используется `motion/react`.
- Для стилизации используется Tailwind CSS v4 с кастомной темой.

## AI-чат

### Реализация

AI-чат запускается на клиенте в `ChatAssistant.tsx`.

Он:
- инициализирует `GoogleGenAI`;
- создаёт chat session;
- передаёт системную инструкцию;
- принимает потоковый ответ модели;
- обрабатывает function call `generatePaymentLink`.

### Поток взаимодействия

```text
Пользователь пишет в чат
  -> frontend отправляет сообщение в Gemini
  -> Gemini отвечает текстом или вызывает функцию
  -> frontend обрабатывает function call
  -> frontend генерирует ссылку на оплату
  -> frontend возвращает результат в модель
  -> модель формирует финальный ответ для пользователя
```

### Ограничения

- AI-чат работает напрямую с Gemini из клиента, а не через защищённый backend proxy.
- Платёжная ссылка собирается на клиенте по шаблону.
- Нет отдельного логирования диалогов или rate limiting.

## Backend

### Точка входа

`server.ts`

Сервер:
- поднимает Express;
- сразу запускает Telegram-бота через `startBot()`;
- регистрирует API endpoints;
- в dev-режиме подключает Vite middleware;
- в production раздаёт `dist`.

### Endpoint'ы

#### `GET /api/health`
Простой health-check.

#### `POST /api/prodamus/pay`
Собирает URL оплаты с параметрами:
- тариф;
- цена;
- контакт;
- имя;
- уникальный `order_id`;
- `customer_extra` с данными пользователя.

#### `POST /api/prodamus/webhook`
Принимает callback от Prodamus.

Текущая логика:
- читает `req.body`;
- проверяет `payment_status === 'success'`;
- извлекает данные клиента и заказа;
- отправляет уведомление в Telegram.

Незавершённая часть:
- полноценная HMAC-проверка подписи не реализована.

#### `POST /api/telegram/notify`
Отправляет в Telegram новую заявку с сайта.

## Telegram-бот

### Точка входа

`bot.ts`

### Хранилище

Используется локальный файл SQLite:

```text
bot.db
```

Создаётся таблица `users` со следующими ключевыми полями:
- `id`
- `username`
- `first_name`
- `step`
- `name`
- `role`
- `pain`
- `feeling`
- `thread_id`
- `audio_sent_at`
- `followup_sent`

### Машина состояний

Основной сценарий выглядит так:

```text
start
  -> ask_name
  -> ask_role
  -> ask_pain
  -> waiting_audio
  -> ask_feeling
  -> finished
```

### Что делает бот

#### В личном чате
- приветствует пользователя;
- собирает анкету;
- отправляет аудио-задание;
- после паузы запрашивает обратную связь;
- предлагает записаться на разбор.

#### В Telegram-группе
- создаёт отдельный forum topic под лида;
- публикует карточку лида;
- маршрутизирует дальнейшую переписку между командой и пользователем.

#### По расписанию
- раз в час ищет пользователей, которым пора отправить follow-up;
- отправляет follow-up тем, у кого прошло 24 часа и `followup_sent = 0`.

### Логика маршрутизации сообщений

```text
Личный чат пользователя
  -> если thread_id есть, сообщения копируются в topic группы

Topic группы
  -> сообщения администратора пересылаются пользователю в личку
```

Это позволяет использовать Telegram-группу как простую CRM-обвязку без отдельной панели.

## Интеграция с Prodamus

Проект использует Prodamus в двух местах:

1. backend генерирует ссылку на оплату;
2. backend принимает webhook после оплаты.

### Поток оплаты

```text
Пользователь выбирает тариф
  -> frontend или AI-чат получает/собирает ссылку оплаты
  -> пользователь оплачивает через Prodamus
  -> Prodamus отправляет webhook
  -> backend уведомляет Telegram
```

### Риски текущей реализации

- нет завершённой проверки подписи webhook;
- часть логики оплаты дублируется между AI-чатом и backend;
- нет явной защиты от повторной обработки одного и того же webhook.

## Интеграция с Telegram

Используются два режима:

1. уведомления от backend через HTTP API Telegram;
2. интерактивный бот через Telegraf.

Backend отправляет:
- лиды;
- уведомления об оплате.

Бот обрабатывает:
- онбординг пользователя;
- двустороннюю переписку;
- follow-up.

### Prodamus

- `PRODAMUS_SECRET_KEY`

### Gemini
- `GEMINI_API_KEY`
- `VITE_GEMINI_API_KEY`

## Сильные стороны текущего решения

- быстрый запуск из одной кодовой базы;
- понятная воронка в Telegram;
- простая интеграция сайта, оплаты и команды продаж;
- минимальная инфраструктурная сложность;
- контент проекта легко редактировать в коде.

## Технический долг

- крупный монолитный `src/App.tsx`;
- отсутствуют тесты;
- отсутствует слой сервисов и модулей для backend;
- секреты частично захардкожены;
- SQLite подходит для MVP, но не для высокой нагрузки и многосерверного режима;
- webhook-верификация требует доработки;
- заглушки в аудио и follow-up ещё не заменены реальными материалами.

## Рекомендуемые следующие шаги

1. Вынести все секреты из кода и проверить `.gitignore`.
2. Завершить верификацию webhook Prodamus через raw body.

4. Разделить `App.tsx` на отдельные секции и компоненты.
5. Добавить журнал событий лидов и оплат.
6. Добавить минимальные интеграционные тесты для API и бота.
7. Подготовить отдельный production-конфиг и инструкции по деплою.
